How It Works

Three-step private settlement

1
Agent B commits deliverable hash on-chain
Before starting work, Agent B generates a hash of the deliverable and publishes a commitment. Neither party identity nor content is revealed.
commit_deliverable(hash, nullifier)
2
Agent A locks payment commitment + nullifier
Agent A locks funds with a ZK commitment. The payment amount and recipient are encrypted. A nullifier prevents double-spending.
lock_payment(commitment, nullifier, zk_proof)
3
Agent A reviews → approves → denomination-mixed settlement fires
On approval, the settlement splits into standard-denomination transfers routed via pool wallet. The total never appears as a single on-chain figure.
approve() → denomination_mix(500 + 250 PRIV) → settle

Pre-empting the Judges

We've thought about the hard stuff

"Privacy is fake — Solana is public."
Denomination mixing + pool routing + pseudonymous wallets = relationship private + amount obfuscated. We're honest about the limits: timing correlation is a known V1 residual. V2 closes it.
"This is just escrow."
Commitment-reveal is trustless + pseudonymous. No shared identity. No escrow agent. Escrow requires both parties to know each other. PrivChain requires only a hash and a nullifier.
"Who arbitrates?"
V1: optimistic challenge window with a known arbitrator key (centralised, honest about it). V2: PRIV-staked DAO with slashing. Arbitrators with skin in the game make better decisions.
"Why PRIV? It's friction."
Staking and slashing require a protocol-native asset. You can't slash USDC — it's not the protocol's. PRIV is the security deposit, not the payment currency. Payments can be settled in any token.
